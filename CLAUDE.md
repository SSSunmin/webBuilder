# 작업 규칙

이 파일은 매 세션 자동 로드됩니다. **짧게 유지하세요** — 항상 켜져 있는 비용이라, 여기엔 *절대 불변규칙*과 *라우터*만 둡니다. 상세는 온디맨드 문서로.

## 행동 원칙

- **위임 우선**: 멀티파일 변경·디버깅·코드리뷰는 서브에이전트(Task)에 위임한다. 한두 줄 사소한 변경은 직접.
- **증거 우선**: "됐다" 전에 실제로 실행/테스트해 결과를 보여준다. 추측으로 완료 선언 금지.
- **작성과 리뷰 분리**: 코드를 짠 그 패스에서 스스로 승인하지 않는다. 리뷰는 별도(`/review` 또는 `code-reviewer`).
- **복잡도에 맞춘 깊이**: 단순 작업은 빠르게, 복잡한 추론만 깊게.
- **불확실하면 표시**: 확신 없는 부분은 추측이라고 명시한다.

## ⛔ 절대 불변규칙 (항상 적용 — 어기면 안 됨)

> 이 프로젝트에서 **절대 어기면 안 되는 규칙**만 5~10줄로 적는다(트리거 없이 항상 지켜야 하는 것). 상세 설명이 필요한 건 여기 말고 `docs/skills/`에 두고 라우터로 가리킨다.
>
> **이 슬롯은 비어 있는 게 정상이다(아래 주석만 있으면 빈 것).** 새 프로젝트라면 **첫 작업을 시작하기 전에**, 슬롯이 비어 있으면 사용자에게 **`/setup-rules` 실행을 먼저 제안**한다 — Claude가 lint·tsconfig·기존 문서·코드 관례를 읽어 후보를 제안하고, **사용자가 확인한 규칙만** 기록한다(추측 금지). 채우기 전까지 이 섹션은 효과가 없으며, 잘못된 규칙이 적용되지 않도록 **예시를 활성 규칙으로 두지 않는다**.

## 📂 온디맨드 문서 라우터 (docs/skills/)

작업 전, 해당 주제 문서를 **먼저 읽고** 작업한다. 항상 로드하지 않고 필요할 때만 — 토큰 절약 + 집중도 유지.

| 작업 종류 | 읽을 문서 |
|---|---|
| (예) UI·스타일·컴포넌트 | `docs/skills/ui/*.md` |
| (예) 피그마 → 코드 (피그마 쓸 때만) | `docs/skills/ui/figma/*.md` |
| (예) 상태·데이터·HTTP | `docs/skills/state/*.md` |
| (예) 구조·환경·컨벤션 | `docs/skills/foundation/*.md` |
| skill/메모 작성 기준 | `docs/skills/guide/skill-guide.md` |

> 새 주제가 반복되면 `docs/skills/`에 문서를 추가하고 이 표에 한 줄 등록한다. 작성법: `docs/skills/guide/skill-guide.md`.

- **(상시) 문서화 제안**: 작업 중 같은 절차를 2회+ 반복했거나, 실수하기 쉬운 패턴·프로젝트 전용 규칙을 발견하면 → 마무리 단계에서 `docs/skills/`에 문서화를 **사용자에게 제안**한다(승인 시 작성, 추측 금지). 자동 생성하지 말고 제안→승인. 기준: `docs/skills/guide/skill-guide.md`.

## ✅ 품질 게이트 (정확도·코드품질)

1. **구현** → 2. **리뷰**(별도 패스 `/review` 또는 `code-reviewer` — critical/major/minor) → 3. **단순화**(선택 `/simplify` 또는 `code-simplifier` — 기능 보존, 중복·네이밍·일관성) → 4. **검증**(`/verify` 또는 통과까지 `/ultraqa`).
- **완료 선언 전 증거 필수**: 테스트/린트/타입체크/실행 결과를 실제로 보여준다. 추측 완료 금지.
- 깨지기 쉬운 게이트(빌드·타입·테스트)는 `/ultraqa --build|--typecheck|--tests`로 통과까지 자동 반복.

## 쓸 수 있는 도구

**서브에이전트(Task로 위임)**: `analyst` · `product-planner` · `impl-planner` · `architect` · `code-reviewer` · `code-simplifier` · `debugger` · `qa-tester` · `knowledge-curator`
사용법 상세 → `USAGE.md`

## 지식 문서(OKF)

- 프로젝트 지식은 `knowledge/` 폴더에 **OKF 번들**로 관리(개념별 `.md` + 프론트매터 `type` 필수, 절대경로 링크).
- DB 스키마·API·아키텍처에 **의미 있는 변경**이 생기면 `/okf`(또는 `knowledge-curator`)로 갱신. 추측 금지, 코드 근거로만.
- 갱신 시 `knowledge/index.md`(목차)·`knowledge/log.md`(이력)도 함께 최신화.
