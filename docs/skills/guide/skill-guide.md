# Skill 문서 & 메모 작성 가이드

## 언제 skill 문서를 만드나 (생성 기준)
하나라도 해당하면 `docs/skills/`에 문서화한다.

- **반복 작업**: 같은 절차를 2회 이상 했거나 앞으로 반복이 확실한 것.
- **복잡한 스캐폴딩**: 여러 파일을 특정 순서로 만들어야 하는 작업.
- **실수하기 쉬운 패턴**: 한 번 틀렸거나 주의가 필요한 절차.
- **프로젝트 전용 규칙**: 이 프로젝트에서만 적용되는 특수 처리.

> **불변규칙 vs skill 문서**: "트리거 없이 항상 지켜야 하는 한 줄짜리"는 루트 `CLAUDE.md`의 *절대 불변규칙*에.
> "어떻게 하는지 절차·예시가 필요한 것"은 여기 `docs/skills/`에. 상세를 CLAUDE.md에 넣지 말 것(항상-로드 비용·희석).

## 작성 형식
[`_TEMPLATE.md`](/docs/skills/_TEMPLATE.md) 복사 → 목적 / 사전조건 / 작업순서 / 예시 / 주의사항 / 관련문서.
- **증거 기반**: 실제 동작하는 최소 예시만. 추측성 코드·존재하지 않는 API 금지.
- **짧고 작업 단위로**: 한 문서가 너무 길면 같은 이름 하위 폴더로 쪼개고 원본은 개요(라우터)로.

## 자동 라우팅(선택)
프론트매터 `name`/`description`을 달면 하버스가 description만 상주시키고 관련 작업에서 본문을 자동으로 펼친다.
→ "작업 전 읽어라"는 소프트 지시 의존을 없애 정확도↑. 토큰은 한 줄만 추가.
description은 **"언제 읽어야 하는지"**를 구체적으로(키워드 포함) 쓴다.

## 새 문서 추가 후
1. 루트 `CLAUDE.md` 라우터 표에 한 줄 등록.
2. 관련 기존 문서의 "관련 문서"에 상호 링크.

## 프로젝트 메모(결정·함정 기록)
코드만 봐선 모르는 배경을 남긴다. 위치는 프로젝트에 맞게(`MEMORY.md` 또는 OKF `knowledge/decisions/`).

기록 기준:
- **결정 사항**: 구조·패턴·라이브러리 선택과 **그 이유**.
- **발견한 버그/함정**: 예상 못한 동작, 주의할 엣지 케이스.
- **되돌린 결정**: 시도했다 원복한 접근과 이유(같은 실수 반복 방지).

원칙: 날짜 기록 / 결론만 말고 **왜**를 남김 / 오래돼 달라지면 `[업데이트]` 표시.
