# 의미 있는 테스트 — 가짜 테스트 방어 + 핵심 로직 판정

## 목적

AI는 **커버리지는 오르지만 아무것도 검증 안 하는** 테스트를 쉽게 짠다. "가짜 테스트 금지"라는 *말*로는 못 막는다 — 막는 건 **테스트를 테스트하는 기계적 장치**다. 이 문서는 ① 가짜 테스트를 잡는 방어선과 ② 비싼 검증을 어디에 집중할지(핵심 로직 기준)를 정한다.

- **트리거**: 테스트 작성(`test-writer`/`/test`), 테스트 리뷰(`code-reviewer`/`/review`).

## A. "항상 통과" 안티패턴 (이러면 가짜)

- **토톨로지** — 구현을 그대로 복붙해 자기 자신과 비교(`expect(add(2,3)).toBe(2+3)`).
- **단언 없음** — 함수를 호출만 하고 `expect`가 없음. 커버리지만 오르고 검증 0.
- **목(mock) 단언** — 실제 동작이 아니라 "목이 불렸다"만 확인. 로직이 틀려도 통과.
- **과도한 모킹** — 진짜 코드가 하나도 안 돌게 막아둠.
- **스냅샷 자동 갱신** — 틀린 출력을 새 스냅샷으로 덮음.
- **조건부 expect** — `if (x) expect(...)` 인데 그 분기가 안 탐.
- `expect(true).toBe(true)` 류.

## B. 방어선 (효과·비용 순 — 쌀수록 먼저)

1. **실패를 먼저 보여라 (see-it-fail) — 가장 쌈, 항상 적용.**
   한 번도 실패한 적 없는 테스트는 의심한다. 테스트를 짠 뒤 **구현을 일부러 깨뜨린(혹은 변경 전) 버전에서 그 테스트가 "올바른 이유로" 실패하는지** 확인하고, 그다음 통과시킨다. 실패를 못 시키는 테스트는 가치가 없다. (특성화 테스트가 이미 이 형태 → [intent-and-pr](/docs/skills/foundation/intent-and-pr.md))

2. **"틀리면 실패하나?" 리뷰 질문 — 쌈.**
   테스트 리뷰의 단 하나의 핵심 질문: **"이 테스트는 대상 코드가 틀렸다면 실패하는가?"** A의 안티패턴 블랙리스트와 대조한다. (작성·리뷰 분리: 짠 사람이 자기 테스트를 같은 패스에서 승인하지 않는다.)

3. **린트 — 자동.**
   단언 누락·스킵·포커스 테스트를 기계로 막는다: `eslint-plugin-jest`(또는 vitest)의 `expect-expect`·`valid-expect`·`no-conditional-expect`·`no-disabled-tests`·`no-focused-tests`. CI/pre-push에서 차단.

4. **뮤테이션 테스트 — 가장 확실, 무거움 → 핵심 로직에만.**
   코드에 일부러 버그(mutant: `+`→`-`, `>`→`>=`, `&&`→`||` 등)를 심고 기존 테스트를 돌린다. 그래도 **통과하면 = 그 버그를 아무도 못 잡는다(테스트 구멍).** 지표는 커버리지 %가 아니라 **mutation score = 죽인 mutant / 전체 mutant** — 단언 없이는 점수가 안 올라 게임이 어렵다.
   - 도구: **Stryker**(JS/TS), **mutmut·cosmic-ray**(Python), **PIT**(Java), **cargo-mutants**(Rust), **gremlins**(Go).
   - 한계: 느리다(mutant마다 스위트 재실행), equivalent mutant(동작이 진짜 같아 못 죽이는 것)는 사람이 거른다, *무엇을* 테스트할지는 안 알려준다.
   - **현실 적용**: 전체 CI 게이트 ❌(느림). **핵심 로직에만**, 처음엔 차단 말고 **리포트**로 돌려 score 낮은 모듈의 테스트를 보강. 익숙해지면 핵심 모듈만 임계치 게이트.

## C. 핵심 로직 판정 — 어디에 강한 검증을 집중하나

**원칙: 핵심 로직 = "조용히 틀렸을 때 비용이 큰 코드".** 버그가 시끄럽게 터지면(크래시) 금방 잡힌다 — 무서운 건 *조용히 잘못된 값*이다.

**결정 게이트 — 3개 질문 중 2개 이상 Yes면 핵심 로직:**

1. **틀리면 누가/무엇이 다치나?** — 돈·데이터 무결성·권한/접근·사용자 신뢰. (피해 범위)
2. **조용히 틀릴 수 있나?** — 분기·경계·상태가 많아 *미묘하게 틀린 값*이 날 수 있나. (미묘함)
3. **되돌리기 어렵나?** — 송금·삭제·외부 전송 *뒤에야* 발견되나. (불가역성)

**거의 항상 핵심인 신호**: 돈·수량 계산(가격·할인·세금·정산·포인트) / 인증·인가 / 데이터 무결성(DB 쓰기·마이그레이션·상태 전이·멱등성) / 신뢰 경계 입력 처리(검증·파싱·새니타이즈) / 분기 있는 비즈니스 규칙 / 불가역·조용한 부수효과 / 널리 의존되는 공용 util·hook.

**핵심이 아닌 것(강한 검증 낭비)**: 순수 표현·마크업·스타일 / 분기 없는 얇은 패스스루 / 설정·상수·타입 / 생성된 코드 / 버그가 즉시 시끄럽게 터지고 고치기 싼 코드.

**직관 분별법**: *"도메인 전문가가 명세를 적어줄 코드"*(핵심) vs *"걱정 없이 다시 생성해도 되는 코드"*(아님).
예) `formatPrice(1234)→"₩1,234"`는 표시용 → 낮음. `calculateFinalPrice(cart, coupons, taxRule)`은 돈·분기·불가역 → 핵심. 둘 다 'price'지만 한쪽만 핵심.

> 보통 핵심 로직은 코드베이스의 10~20%다. 거기에만 뮤테이션 같은 비싼 검증을 집중한다.

## 주의사항 / 함정

- 커버리지 100%는 "실행됐다"일 뿐 "검증됐다"가 아니다 — 가짜 테스트도 100%가 나온다. 신뢰 지표는 mutation score.
- 파일명·폴더로 핵심을 정하지 말 것 — **"조용히 틀리면 비용이 큰가"**로 함수 단위 판정.
- 뮤테이션을 전체에 매번 돌리지 말 것(느려서 CI가 막힘). 핵심 한정 + 리포트 우선.
