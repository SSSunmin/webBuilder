# 의도 보존 — 작은 변경 + "왜" 기록

## 목적

AI가 코드를 대량 생산하면 **작성은 싸지지만 이해는 비싸진다.** 추론은 거쳤어도 결과 코드만 남아, 리뷰어가 "왜 이렇게 했는지"를 처음부터 역설계해야 한다. 그 비용을 리뷰 단에서 복구하는 것보다, **만들 때 의도를 함께 남기는 게 훨씬 싸다.**

- **트리거**: 커밋/PR 생성 시, 비자명한 설계 선택(대안을 버린 결정)을 코드에 반영할 때.

## 작업 순서

### 1. 변경은 작게 — 한 PR = 한 의도

- 리팩터링과 기능 추가를 **한 PR에 섞지 않는다.** "겸사겸사" 주변 코드를 건드리지 않는다(게으른 시니어: 요청 안 한 변경 금지).
- diff가 커지면 멈추고 쪼갤 수 있는지 본다. 대략 **±400줄을 넘으면** PR 설명에 "왜 큰지" 한 줄을 남긴다.

**라인 수가 아니라 리뷰 부담을 줄인다 — 생성/포맷/리네임은 별도 커밋으로 분리:**

- 자동 생성물(스냅샷·lock·마이그레이션·codegen), 포맷팅, 단순 리네임/이동은 **로직 변경과 다른 커밋**으로 떼어낸다.
- 그러면 리뷰어가 커밋 제목만 보고 **"이 커밋은 안 읽어도 됨"**을 바로 안다. 라인 수가 1000줄이어도 *읽어야 할* 줄은 작아진다 — **PR 크기 = 리뷰 부담이 아니다.**
- 커밋 제목에 성격을 드러낸다: `chore: prettier 포맷`, `refactor: rename Foo→Bar (no behavior change)`, `chore: 마이그레이션 자동 생성`.

### 2. 코드에 "왜"를 남긴다 — `// why:`

- 비자명한 설계 선택에는 **무엇(what)이 아니라 왜(why)**를 짧게 남긴다. 코드가 *무엇을* 하는지는 코드가 말한다 — 주석은 *왜 이 방식인지, 어떤 대안을 왜 버렸는지*를 적는다.

```ts
// why: Map이 아니라 배열 — N<10이라 순회가 더 싸고, 삽입 순서가 의미를 가진다
const items = []
```

- 의도적 단순화·한계는 기존 규칙대로 `ponytail:` 주석으로 한계와 업그레이드 경로를 남긴다.

### 3. 커밋 메시지 — 제목 + 본문에 "왜"

- 제목은 명령형 한 줄. **본문 1줄 이상에 "왜"**(배경/문제/결정 이유)를 적는다. 한 줄짜리 커밋으로 끝내지 않는다.
- 서명·`Co-Authored-By`·Claude 관련 trailer 금지(루트 CLAUDE.md 불변규칙).

### 4. PR 설명 — 아래 5칸을 채운다

빈칸이 어색하게 보이도록 항상 채운다. **Why**와 **검토 주의점**이 의도 소실을 막는 핵심 칸이다.

```markdown
## 무엇을 (What)
## 왜 (Why) — 이 변경이 필요한 배경/문제
## 어떻게 (How) — 핵심 설계 선택과 그 이유(버린 대안 포함)
## 검증 (How verified) — 실행/테스트 결과(증거 우선)
## 검토 시 주의점 — 리뷰어가 의심해야 할 부분
```

## GitHub 레포에 적용할 때 (선택 — 강제 장치)

이 킷을 쓰는 프로젝트가 git/GitHub 레포라면, 사람의 선의가 아니라 장치로 거든다. **크기는 차단 사유가 아니다 — 대화를 트는 신호일 뿐.** 아래를 직접 만들어도 되고, **`/setup-gates`를 실행하면 스택을 감지해 물어보고 실제 파일을 설치**해 준다. 가벼운 것부터:

1. **PR 템플릿** — 위 5칸을 `.github/PULL_REQUEST_TEMPLATE.md`로 저장(마찰 0, 효과 큼). 여기부터 시작.
2. **PR 크기는 차단이 아니라 라벨/코멘트** — `pull_request` 워크플로에서 추가/삭제 라인을 세어 임계치를 넘으면 `size/large` **라벨을 붙이고 한 번만 코멘트**한다(차단 X). 코멘트 문구 예:
   > 이 PR은 큽니다. ① 쪼갤 수 있나요? ② 없다면 왜 한 덩어리여야 하는지 한 줄 적어주세요.
3. **why-big이 정당하면 그냥 통과** — 작성자가 정당한 사유(예: "마이그레이션 자동 생성", "단일 기능, 쪼개면 빌드 깨짐", "포맷/리네임은 별도 커밋으로 이미 분리")를 PR 본문에 한 줄 남기면 **그대로 머지한다.** 크기 자체로 막지 않는다.

> 목표는 "큰 PR 금지"가 아니라 *큰 PR에는 이유가 적혀 있게* 하는 것. hard-block(머지 차단)은 팀이 충분히 익숙해진 뒤, 그것도 "크기"가 아니라 "Why 칸이 통째로 비었을 때"에만 신중히 켠다.

## 주의사항 / 함정

- **AI 리뷰 도구를 여러 개 까는 것으로 갈음하지 말 것.** 사라진 의도를 *리뷰 단*에서 복구하려 하면 노이즈·알람 피로만 늘고 "아무도 진짜로는 안 보는" 상태가 된다. 가장 싼 해법은 *애초에 작고 의도가 적힌 변경*이다.
- 주석에 *무엇*을 적지 말 것 — 코드와 중복되고 금방 낡는다. 오직 *왜*만.
