`docs/BACKLOG.md` 기반 자율 루프. `/loop`(간격 생략, self-paced)으로 돌린다. **한 회차 = 작업 단위 하나**가 원칙이다.
이 프로젝트의 서브에이전트·커맨드·품질 게이트를 적극 활용한다(맨몸으로 하지 않는다). 이 루프는 `docs/BACKLOG.md`의 **"진행 규칙"을 자동화**한 것이다.

> **백로그의 항목 = 승인된 계획.** 완료 기준이 명확한 항목은 진행한다. 단, 완료 기준이 모호하거나 "연기/Stage B(선행 설계 필요)"면 멈추고 사람에게 묻는다(추측 금지).

## 매 회차 할 일

1. **다음 작업 선택** — `docs/BACKLOG.md`를 읽어, 표에서 **위에서부터 상태가 `완료`가 아니고 다음 작업 단위의 완료 기준이 명확한 첫 항목**(또는 그 항목의 다음 Stage)을 고른다.
   - 그런 항목이 없으면(전부 완료이거나 남은 게 전부 "연기/설계 선행") → "진행 가능한 작업 없음 — 루프 종료" 한 줄 출력 후 **다음 회차를 잡지 않는다.** 필요하면 사람에게 "다음에 뭘 할지" 묻는다.
   - 완료 기준이 비었거나 모호하면 → 구현하지 말고 **멈추고 묻는다.**
   - `git status`가 더러우면(미커밋 변경) 멈추고 알린다 — 남의 작업 위에 쌓지 않는다.

2. **브랜치 분리 (스택형)** — `task/<번호>-<짧은슬러그>` 브랜치 생성. **분기 기준(base)은 직전에 완료한 작업의 브랜치**(스택 — 머지 안 해도 의존 작업이 이어짐). 첫 작업이거나 무관하면 `main`에서 분기. **`main` 직접 작업·커밋 금지.** `docs/BACKLOG.md`의 해당 항목 상태를 "진행 중"으로 표시.

3. **계획(필요 시) + 구현** — 멀티파일/비자명하면 `impl-planner`로 짧게 설계 후 진행. 구현은 **위임 우선**(멀티파일·디버깅은 서브에이전트), 한두 줄은 직접. 게으른 시니어 — 완료 기준을 만족하는 **최소 변경**만. 비자명한 선택엔 `// why:`.

4. **품질 게이트 (CLAUDE.md "품질 게이트" 순서)**
   - **리뷰**: `/review`(또는 `code-reviewer`) — 의도 일치·동작 보존·버그·보안 체크리스트 대조. 작성·리뷰 분리. critical/major 해소.
   - **테스트**: 분기 로직이면 `/test`(`test-writer`) — see-it-fail + 행복/엣지/에러. 동작 보존이 목표면 특성화 테스트 먼저. 기준 `docs/skills/foundation/test-quality.md`. (이 프로젝트 테스트 = `vitest run`.)
   - **검증**: `/ultraqa`로 `--tests`·`--typecheck`·`--build` 중 해당분이 **통과할 때까지** 자동 반복. (명령: `vitest run` / `tsc -b` / `vite build` / `eslint .`)
   - **막히면 중단**: `/ultraqa`가 5회/동일실패 3회로 중단 → 항목을 "막힘"으로 표시하고 사유를 적고, **커밋하지 말고** 보고 후 회차 종료.

5. **OKF 지식 갱신 (해당 작업만)** — DB 스키마·API·아키텍처에 **의미 있는 변경**이면 `/okf`(또는 `knowledge-curator`)로 `knowledge/` 갱신(코드 근거만, 추측 금지). `knowledge/index.md`·`log.md`도 함께. UI·스타일·단순 변경 등 지식에 영향 없으면 건너뛴다. 갱신분은 **같은 브랜치에 별도 커밋**(`docs(okf): …`).

6. **커밋 + PR (게이트 초록일 때만 — 머지는 사람)**
   - 변경분만 커밋(제목 + 본문에 "왜", **서명/Co-Authored-By 금지**).
   - 원격 + `gh` 있으면 → push하고 **PR 생성**. **PR base는 2번에서 분기한 브랜치**(스택 PR). 본문: What/Why/How/검증 결과/검토 주의점. **머지하지 않는다.**
   - 원격/`gh` 없으면 → 로컬 커밋까지만.

7. **백로그 갱신** — `docs/BACKLOG.md`의 상태 표 + 해당 섹션을 갱신(완료 표시 + 무엇을 했는지 + 검증 증거). 옆에 **브랜치명 + 커밋/PR**을 적어 다음 작업이 base로 쓰게 한다.

8. **보고 + 다음** — 한 줄 보고(무엇을·검증 결과·OKF 갱신 여부·PR). 진행 가능한 작업이 더 있으면 짧은 간격으로 다음, 없으면 종료(1번).

## 머지 (사람 몫)

스택 PR이라 **아래부터 순서대로 머지**(#1 → #2 → …). 맨 아래를 main에 머지하면 GitHub가 위 PR의 base를 자동 조정. 루프는 머지를 기다리지 않으므로 머지 안 해도 작업은 계속 쌓인다.

## 절대 규칙 (안전장치)

- **머지·force push·브랜치 삭제·파일 삭제 금지.** 보호 브랜치 직접 push 금지.
- **한 회차 = 작업 하나.** 몰아 처리 금지(회차가 곧 사람 체크포인트).
- 증거 우선 — "통과"는 실제 명령 출력으로만. 추측 완료 금지.
- 모호/위험/불가역이거나 게이트가 막히면 → **멈추고 보고.** 밀어붙이지 않는다.
- 커밋·PR 외 외부 발신(배포·삭제 등) 금지.
