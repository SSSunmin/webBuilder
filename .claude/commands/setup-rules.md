`CLAUDE.md`의 "⛔ 절대 불변규칙" 슬롯을 채운다. **planner처럼 질문으로 확정**하고, 추측으로 적지 않는다.
(서브에이전트는 사용자에게 직접 못 물으므로 **이 인터뷰는 메인 Claude가 진행**한다. 무거운 스캔만 읽기전용 에이전트에 위임.)

진행:

1. **현재 상태 확인** — `CLAUDE.md`의 불변규칙 섹션이 비어 있는지(주석/안내만 있는지) 본다.
   - 비어 있으면 → 신규 셋업. 이미 규칙이 있으면 → 보강/수정할지 사용자에게 먼저 묻는다.

2. **근거 수집 (코드 기반, 추측 금지)** — 다음을 읽어 "항상 지켜야 하는 규칙" 후보를 뽑는다. 파일이 많으면 읽기전용 에이전트(`architect`/`impl-planner`)에 스캔을 위임해도 된다:
   - lint/포맷 설정(`.eslintrc*`, `.prettierrc*`), `tsconfig.json`, `package.json` scripts
   - 기존 `CLAUDE.md`/`README`/`docs` 규칙, 디자인 토큰(`tokens.css`/`@theme`), 폴더·네이밍 컨벤션
   - git 관례(브랜치/커밋 메시지 패턴) — 있으면

3. **질문으로 확정 (planner식)** — 후보를 사용자에게 제시하고 `AskUserQuestion`으로 확인한다. **코드만으론 알 수 없는 것은 반드시 묻는다**:
   - 브랜치/커밋 정책(`main`/`develop` 직접 커밋 금지? 커밋 서명 금지? 커밋은 요청 시에만?)
   - 디자인·단위 규칙(예: `1rem = N px`, 클래스 네이밍 케이스)
   - 절대 쓰면 안 되는 라이브러리/패턴
   - 보안·배포 금지사항(예: 시크릿 커밋 금지, 특정 명령 금지)

4. **승인된 것만 기록** — 사용자가 확정한 규칙만 `CLAUDE.md`의 "⛔ 절대 불변규칙" 섹션에 **5~10줄**로 적는다(안내 주석은 지움). 상세 설명이 필요한 건 규칙으로 적지 말고 `docs/skills/`로 빼고 라우터 표에 등록.

5. **요약 보고** — 무엇을 근거로 무엇을 적었는지 / 사용자가 보류·삭제한 건 무엇인지 / docs/skills로 보낸 건 무엇인지.

원칙:
- **추측 금지** — 코드 근거 + 사용자 승인. 확실치 않으면 적지 말고 묻는다.
- **짧게** — 항상-로드 비용이라 5~10줄. 트리거 없이 늘 지켜야 하는 것만(절차·예시는 `docs/skills/`).
- 다 끝나면 이 슬롯은 "그 프로젝트의 진짜 불변규칙"만 남는다.
